The Data Gateway is the HTTP-based bridge between instant3Dhub and wherever your 3D files actually live. Rather than pushing your CAD data into a proprietary storage layer, instant3Dhub resolves 3D resources by URN and delegates all actual data retrieval to the Data Gateway. This keeps your existing data infrastructure in place while giving instant3Dhub a consistent, protocol-neutral interface to work with.

## How the Data Gateway Works

When a client requests a 3D model, instant3Dhub receives a resource URN — a stable, opaque identifier like `urn:company:models:assembly-4711`. The hub hands that URN to the Data Gateway, which consults its **URN Mapper** configuration to translate the URN into a concrete HTTP URL. It then fetches the resource over HTTP(S) and streams the response back to the hub.

```text
instant3Dhub Core
      │  requests URN
      ▼
 Data Gateway
      │  resolves URN → URL via URN Mapper
      ▼
 HTTP(S) Backend  (PLM, file server, blob store, …)
```

Because the Data Gateway speaks plain HTTP, **any existing HTTP server is a valid backend out of the box** — no agent software or plugin required on the data source side. You configure the translation logic entirely within instant3Dhub.

## URN Mapper Configuration

The URN Mapper lives in your instant3Dhub configuration (or Helm `values.yaml`) and contains one or more prefix-to-URL mappings. When a URN arrives, the gateway finds the longest matching prefix and substitutes the prefix with the corresponding `baseUrl`.

```yaml
urnMapper:
  mappings:
    - urnPrefix: "urn:company:models:"
      baseUrl: "https://plm.example.com/api/3d/"
    - urnPrefix: "urn:company:legacy:"
      baseUrl: "https://files.example.com/3d/"
```

For example, `urn:company:models:assembly-4711` resolves to `https://plm.example.com/api/3d/assembly-4711`.

<Tip>
  Order your mappings from most specific to least specific. The gateway matches the longest prefix first, so more-specific patterns always win.
</Tip>

You can define as many mappings as you need. Each mapping can point to a different host, port, path prefix, or authentication scheme.

## Authentication

If your backend requires authentication, the Data Gateway forwards the `Authorization` header from the incoming request directly to the backend. This means:

- **Bearer token pass-through** — the end user's token is forwarded automatically, so backend access controls are respected without any extra configuration.
- **Service-account authentication** — for machine-to-machine scenarios where a dedicated service account authenticates on behalf of all users, the gateway can inject credentials on a per-mapping basis. The exact configuration syntax for header injection, OAuth 2.0 token exchange, and credential storage depends on your instant3Dhub version. Refer to the reference implementation packages available on the [support portal](https://support.threedy.io) for your specific backend.

<Warning>
  Never hard-code credentials directly in your `values.yaml` when storing it in version control. Use Kubernetes Secrets and reference them via environment variables in your gateway configuration.
</Warning>

## Signed Input Configuration

Signed Input Configuration lets you create cryptographically signed references to specific resources. Instead of passing a raw URN, a client presents a signed token that encodes the URN, an optional access scope, and an expiry time. The Data Gateway verifies the signature before resolving the resource, preventing unauthorized access to URNs that a client did not receive explicitly.

Enabling signed inputs requires a signing key to be configured in your instant3Dhub deployment. Consult the [Data Gateway API reference](/v3.14/api/data-gateway-api) and the [support portal](https://support.threedy.io) for the configuration values applicable to your version, including key algorithm options and token generation utilities.

## Caching

The Data Gateway supports response caching to reduce repeated round-trips to your PLM or file backend. Caching is especially valuable when many users load the same large assembly simultaneously. Cache size and TTL are tunable deployment parameters — consult the support portal at [support.threedy.io](https://support.threedy.io) for the specific configuration keys for your instant3Dhub version.

<Note>
  Cached entries are keyed by the resolved URL. If your backend uses query-string tokens for authentication, review the caching documentation to ensure those tokens are excluded from the cache key, or your cache hit rate will be very low.
</Note>

## Enabling TLS Between the Hub and the Gateway

By default, instant3Dhub and the Data Gateway communicate over the cluster-internal network. If your gateway is deployed outside the cluster — for example, on a separate host or behind a corporate proxy — you can configure TLS for hub-to-gateway traffic by supplying a gateway endpoint URL and a CA certificate for verification. Contact [support@threedy.io](mailto:support@threedy.io) for the configuration values applicable to your deployment.

## Testing Your Configuration

After applying your configuration, use the instant3Dhub health endpoint to confirm the gateway can reach each backend:

```bash
curl https://hub.example.com/health/gateway
```

A successful response lists each configured URN prefix alongside its resolved status:

```json
{
  "urn:company:models:": { "status": "ok", "latencyMs": 12 },
  "urn:company:legacy:": { "status": "ok", "latencyMs": 45 }
}
```

If a backend is unreachable, the status shows `"error"` with a diagnostic message to help you troubleshoot the connection.
